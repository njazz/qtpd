00014 /*
00015    PROPOSAL: A Generic Python Object Interface for Python C Modules
00016 
00017 Problem
00018 
00019   Python modules written in C that must access Python objects must do
00020   so through routines whose interfaces are described by a set of
00021   include files.  Unfortunately, these routines vary according to the
00022   object accessed.  To use these routines, the C programmer must check
00023   the type of the object being used and must call a routine based on
00024   the object type.  For example, to access an element of a sequence,
00025   the programmer must determine whether the sequence is a list or a
00026   tuple:
00027 
00028     if(is_tupleobject(o))
00029       e=gettupleitem(o,i)
00030     else if(is_listitem(o))
00031       e=getlistitem(o,i)
00032 
00033   If the programmer wants to get an item from another type of object
00034   that provides sequence behavior, there is no clear way to do it
00035   correctly.
00036 
00037   The persistent programmer may peruse object.h and find that the
00038   _typeobject structure provides a means of invoking up to (currently
00039   about) 41 special operators.  So, for example, a routine can get an
00040   item from any object that provides sequence behavior. However, to
00041   use this mechanism, the programmer must make their code dependent on
00042   the current Python implementation.
00043 
00044   Also, certain semantics, especially memory management semantics, may
00045   differ by the type of object being used.  Unfortunately, these
00046   semantics are not clearly described in the current include files.
00047   An abstract interface providing more consistent semantics is needed.
00048 
00049 Proposal
00050 
00051   I propose the creation of a standard interface (with an associated
00052   library of routines and/or macros) for generically obtaining the
00053   services of Python objects.  This proposal can be viewed as one
00054   components of a Python C interface consisting of several components.
00055 
00056   From the viewpoint of C access to Python services, we have (as
00057   suggested by Guido in off-line discussions):
00058 
00059   - "Very high level layer": two or three functions that let you exec or
00060     eval arbitrary Python code given as a string in a module whose name is
00061     given, passing C values in and getting C values out using
00062     mkvalue/getargs style format strings.  This does not require the user
00063     to declare any variables of type "PyObject *".  This should be enough
00064     to write a simple application that gets Python code from the user,
00065     execs it, and returns the output or errors.  (Error handling must also
00066     be part of this API.)
00067 
00068   - "Abstract objects layer": which is the subject of this proposal.
00069     It has many functions operating on objects, and lest you do many
00070     things from C that you can also write in Python, without going
00071     through the Python parser.
00072 
00073   - "Concrete objects layer": This is the public type-dependent
00074     interface provided by the standard built-in types, such as floats,
00075     strings, and lists.  This interface exists and is currently
00076     documented by the collection of include files provided with the
00077     Python distributions.
00078 
00079   From the point of view of Python accessing services provided by C
00080   modules:
00081 
00082   - "Python module interface": this interface consist of the basic
00083     routines used to define modules and their members.  Most of the
00084     current extensions-writing guide deals with this interface.
00085 
00086   - "Built-in object interface": this is the interface that a new
00087     built-in type must provide and the mechanisms and rules that a
00088     developer of a new built-in type must use and follow.
00089 
00090   This proposal is a "first-cut" that is intended to spur
00091   discussion. See especially the lists of notes.
00092 
00093   The Python C object interface will provide four protocols: object,
00094   numeric, sequence, and mapping.  Each protocol consists of a
00095   collection of related operations.  If an operation that is not
00096   provided by a particular type is invoked, then a standard exception,
00097   NotImplementedError is raised with a operation name as an argument.
00098   In addition, for convenience this interface defines a set of
00099   constructors for building objects of built-in types.  This is needed
00100   so new objects can be returned from C functions that otherwise treat
00101   objects generically.
00102 
00103 Memory Management
00104 
00105   For all of the functions described in this proposal, if a function
00106   retains a reference to a Python object passed as an argument, then the
00107   function will increase the reference count of the object.  It is
00108   unnecessary for the caller to increase the reference count of an
00109   argument in anticipation of the object's retention.
00110 
00111   All Python objects returned from functions should be treated as new
00112   objects.  Functions that return objects assume that the caller will
00113   retain a reference and the reference count of the object has already
00114   been incremented to account for this fact.  A caller that does not
00115   retain a reference to an object that is returned from a function
00116   must decrement the reference count of the object (using
00117   DECREF(object)) to prevent memory leaks.
00118 
00119   Note that the behavior mentioned here is different from the current
00120   behavior for some objects (e.g. lists and tuples) when certain
00121   type-specific routines are called directly (e.g. setlistitem).  The
00122   proposed abstraction layer will provide a consistent memory
00123   management interface, correcting for inconsistent behavior for some
00124   built-in types.
