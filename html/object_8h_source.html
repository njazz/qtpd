00009 
00010 /*
00011 Objects are structures allocated on the heap.  Special rules apply to
00012 the use of objects to ensure they are properly garbage-collected.
00013 Objects are never allocated statically or on the stack; they must be
00014 accessed through special macros and functions only.  (Type objects are
00015 exceptions to the first rule; the standard types are represented by
00016 statically initialized type objects, although work on type/class unification
00017 for Python 2.2 made it possible to have heap-allocated type objects too).
00018 
00019 An object has a 'reference count' that is increased or decreased when a
00020 pointer to the object is copied or deleted; when the reference count
00021 reaches zero there are no references to the object left and it can be
00022 removed from the heap.
00023 
00024 An object has a 'type' that determines what it represents and what kind
00025 of data it contains.  An object's type is fixed when it is created.
00026 Types themselves are represented as objects; an object contains a
00027 pointer to the corresponding type object.  The type itself has a type
00028 pointer pointing to the object representing the type 'type', which
00029 contains a pointer to itself!).
00030 
00031 Objects do not float around in memory; once allocated an object keeps
00032 the same size and address.  Objects that must hold variable-size data
00033 can contain pointers to variable-size parts of the object.  Not all
00034 objects of the same type have the same size; but the size cannot change
00035 after allocation.  (These restrictions are made so a reference to an
00036 object can be simply a pointer -- moving an object would require
00037 updating all the pointers, and changing an object's size would require
00038 moving it if there was another object right next to it.)
00039 
00040 Objects are always accessed through pointers of the type 'PyObject *'.
00041 The type 'PyObject' is a structure that only contains the reference count
00042 and the type pointer.  The actual memory allocated for an object
00043 contains other data that can only be accessed after casting the pointer
00044 to a pointer to a longer structure type.  This longer type must start
00045 with the reference count and type fields; the macro PyObject_HEAD should be
00046 used for this (to accommodate for future changes).  The implementation
00047 of a particular object type can cast the object pointer to the proper
00048 type and back.
00049 
00050 A standard interface exists for objects that contain an array of items
00051 whose size is determined when the object is allocated.
00052 */

00538 /*
00539 `Type flags (tp_flags)
00540 
00541 These flags are used to extend the type structure in a backwards-compatible
00542 fashion. Extensions can use the flags to indicate (and test) when a given
00543 type structure contains a new feature. The Python core will use these when
00544 introducing new functionality between major revisions (to avoid mid-version
00545 changes in the PYTHON_API_VERSION).
00546 
00547 Arbitration of the flag bit positions will need to be coordinated among
00548 all extension writers who publically release their extensions (this will
00549 be fewer than you might expect!)..
00550 
00551 Python 1.5.2 introduced the bf_getcharbuffer slot into PyBufferProcs.
00552 
00553 Type definitions should use Py_TPFLAGS_DEFAULT for their tp_flags value.
00554 
00555 Code can use PyType_HasFeature(type_ob, flag_value) to test whether the
00556 given type object has a specified feature.

00664 The macros Py_INCREF(op) and Py_DECREF(op) are used to increment or decrement
00665 reference counts.  Py_DECREF calls the object's deallocator function when
00666 the refcount falls to 0; for
00667 objects that don't contain references to other objects or heap memory
00668 this can be the standard function free().  Both macros can be used
00669 wherever a void expression is allowed.  The argument must not be a
00670 NULL pointer.  If it may be NULL, use Py_XINCREF/Py_XDECREF instead.
00671 The macro _Py_NewReference(op) initialize reference counts to 1, and
00672 in special builds (Py_REF_DEBUG, Py_TRACE_REFS) performs additional
00673 bookkeeping appropriate to the special build.

00675 We assume that the reference count field can never overflow; this can
00676 be proven when the size of the field is the same as the pointer size, so
00677 we ignore the possibility.  Provided a C int is at least 32 bits (which
00678 is implicitly assumed in many parts of this code), that's enough for

00683 XXX can and should be deallocated.

00780 /* Safely decref `op` and set `op` to NULL, especially useful in tp_clear
00781  * and tp_dealloc implementatons.
00782  *
00783  * Note that "the obvious" code can be deadly:
00784  *
00785  *     Py_XDECREF(op);
00786  *     op = NULL;
00787  *
00788  * Typically, `op` is something like self->containee, and `self` is done
00789  * using its `containee` member.  In the code sequence above, suppose
00790  * `containee` is non-NULL with a refcount of 1.  Its refcount falls to
00791  * 0 on the first line, which can trigger an arbitrary amount of code,
00792  * possibly including finalizers (like __del__ methods or weakref callbacks)
00793  * coded in Python, which in turn can release the GIL and allow other threads
00794  * to run, etc.  Such code may even invoke methods of `self` again, or cause
00795  * cyclic gc to trigger, but-- oops! --self->containee still points to the
00796  * object being torn down, and it may be in an insane state while being torn
00797  * down.  This has in fact been a rich historic source of miserable (rare &
00798  * hard-to-diagnose) segfaulting (and other) bugs.
00799  *
00800  * The safe way is:
00801  *
00802  *      Py_CLEAR(op);
00803  *
00804  * That arranges to set `op` to NULL _before_ decref'ing, so that any code
00805  * triggered as a side-effect of `op` getting torn down no longer believes
00806  * `op` points to a valid object.
00807  *
00808  * There are cases where it's safe to use the naive code, but they're brittle.
00809  * For example, if `op` points to a Python integer, you know that destroying
00810  * one of those can't cause problems -- but in part that relies on that
00811  * Python integers aren't currently weakly referencable.  Best practice is
00812  * to use Py_CLEAR() even if you can't think of a reason for why you need to.
00813  */
00814 #define Py_CLEAR(op)                            \
00815     do {                                        \
00816         if (op) {                               \
00817             PyObject *_py_tmp = (PyObject *)(op);               \
00818             (op) = NULL;                        \
00819             Py_DECREF(_py_tmp);                 \
00820         }                                       \
00821     } while (0)

00825 #define Py_XDECREF(op) do { if ((op) == NULL) ; else Py_DECREF(op); } while (0)

00882 /*
00883 More conventions
00884 ================
00885 
00886 Argument Checking
00887 -----------------
00888 
00889 Functions that take objects as arguments normally don't check for nil
00890 arguments, but they do check the type of the argument, and return an
00891 error if the function doesn't apply to the type.
00892 
00893 Failure Modes
00894 -------------
00895 
00896 Functions may fail for a variety of reasons, including running out of
00897 memory.  This is communicated to the caller in two ways: an error string
00898 is set (see errors.h), and the function result differs: functions that
00899 normally return a pointer return NULL for failure, functions returning
00900 an integer return -1 (which could be a legal return value too!), and
00901 other functions return 0 for success and -1 for failure.
00902 Callers should always check for errors before using the result.  If
00903 an error was set, the caller must either explicitly clear it, or pass
00904 the error on to its caller.
00905 
00906 Reference Counts
00907 ----------------
00908 
00909 It takes a while to get used to the proper usage of reference counts.
00910 
00911 Functions that create an object set the reference count to 1; such new
00912 objects must be stored somewhere or destroyed again with Py_DECREF().
00913 Some functions that 'store' objects, such as PyTuple_SetItem() and
00914 PyList_SetItem(),
00915 don't increment the reference count of the object, since the most
00916 frequent use is to store a fresh object.  Functions that 'retrieve'
00917 objects, such as PyTuple_GetItem() and PyDict_GetItemString(), also
00918 don't increment
00919 the reference count, since most frequently the object is only looked at
00920 quickly.  Thus, to retrieve an object and store it again, the caller
00921 must call Py_INCREF() explicitly.
00922 
00923 NOTE: functions that 'consume' a reference count, like
00924 PyList_SetItem(), consume the reference even if the object wasn't
00925 successfully stored, to simplify error handling.
00926 
00927 It seems attractive to make other functions that take an object as
00928 argument consume a reference count; however, this may quickly get
00929 confusing (even the current practice is already confusing).  Consider
00930 it carefully, it may save lots of calls to Py_INCREF() and Py_DECREF() at
00931 times.
00932 */

00935 /* Trashcan mechanism, thanks to Christian Tismer.
00936 
00937 When deallocating a container object, it's possible to trigger an unbounded
00938 chain of deallocations, as each Py_DECREF in turn drops the refcount on "the
00939 next" object in the chain to 0.  This can easily lead to stack faults, and
00940 especially in threads (which typically have less stack space to work with).
00941 
00942 A container object that participates in cyclic gc can avoid this by
00943 bracketing the body of its tp_dealloc function with a pair of macros:
00944 
00945 static void
00946 mytype_dealloc(mytype *p)
00947 {
00948     ... declarations go here ...
00949 
00950     PyObject_GC_UnTrack(p);        // must untrack first
00951     Py_TRASHCAN_SAFE_BEGIN(p)
00952     ... The body of the deallocator goes here, including all calls ...
00953     ... to Py_DECREF on contained objects.                         ...
00954     Py_TRASHCAN_SAFE_END(p)
00955 }
00956 
00957 CAUTION:  Never return from the middle of the body!  If the body needs to
00958 "get out early", put a label immediately before the Py_TRASHCAN_SAFE_END
00959 call, and goto it.  Else the call-depth counter (see below) will stay
00960 above 0 forever, and the trashcan will never get emptied.
00961 
00962 How it works:  The BEGIN macro increments a call-depth counter.  So long
00963 as this counter is small, the body of the deallocator is run directly without
00964 further ado.  But if the counter gets large, it instead adds p to a list of
00965 objects to be deallocated later, skips the body of the deallocator, and
00966 resumes execution after the END macro.  The tp_dealloc routine then returns
00967 without deallocating anything (and so unbounded call-stack depth is avoided).
00968 
00969 When the call stack finishes unwinding again, code generated by the END macro
00970 notices this, and calls another routine to deallocate all the objects that
00971 may have been added to the list of deferred deallocations.  In effect, a
00972 chain of N deallocations is broken into N / PyTrash_UNWIND_LEVEL pieces,
00973 with the call stack never exceeding a depth of PyTrash_UNWIND_LEVEL.
00974 */
00975
