00044 /*
00045  *       Define macros for handling SIGFPE.
00046  *       Lee Busby, LLNL, November, 1996
00047  *       busby1@llnl.gov
00048  * 
00049  *********************************************
00050  * Overview of the system for handling SIGFPE:
00051  * 
00052  * This file (Include/pyfpe.h) defines a couple of "wrapper" macros for
00053  * insertion into your Python C code of choice. Their proper use is
00054  * discussed below. The file Python/pyfpe.c defines a pair of global
00055  * variables PyFPE_jbuf and PyFPE_counter which are used by the signal
00056  * handler for SIGFPE to decide if a particular exception was protected
00057  * by the macros. The signal handler itself, and code for enabling the
00058  * generation of SIGFPE in the first place, is in a (new) Python module
00059  * named fpectl. This module is standard in every respect. It can be loaded
00060  * either statically or dynamically as you choose, and like any other
00061  * Python module, has no effect until you import it.
00062  * 
00063  * In the general case, there are three steps toward handling SIGFPE in any
00064  * Python code:
00065  * 
00066  * 1) Add the *_PROTECT macros to your C code as required to protect
00067  *    dangerous floating point sections.
00068  * 
00069  * 2) Turn on the inclusion of the code by adding the ``--with-fpectl''
00070  *    flag at the time you run configure.  If the fpectl or other modules
00071  *    which use the *_PROTECT macros are to be dynamically loaded, be
00072  *    sure they are compiled with WANT_SIGFPE_HANDLER defined.
00073  * 
00074  * 3) When python is built and running, import fpectl, and execute
00075  *    fpectl.turnon_sigfpe(). This sets up the signal handler and enables
00076  *    generation of SIGFPE whenever an exception occurs. From this point
00077  *    on, any properly trapped SIGFPE should result in the Python
00078  *    FloatingPointError exception.
00079  * 
00080  * Step 1 has been done already for the Python kernel code, and should be
00081  * done soon for the NumPy array package.  Step 2 is usually done once at
00082  * python install time. Python's behavior with respect to SIGFPE is not
00083  * changed unless you also do step 3. Thus you can control this new
00084  * facility at compile time, or run time, or both.
00085  * 
00086  ******************************** 
00087  * Using the macros in your code:
00088  * 
00089  * static PyObject *foobar(PyObject *self,PyObject *args)
00090  * {
00091  *     ....
00092  *     PyFPE_START_PROTECT("Error in foobar", return 0)
00093  *     result = dangerous_op(somearg1, somearg2, ...);
00094  *     PyFPE_END_PROTECT(result)
00095  *     ....
00096  * }
00097  * 
00098  * If a floating point error occurs in dangerous_op, foobar returns 0 (NULL),
00099  * after setting the associated value of the FloatingPointError exception to
00100  * "Error in foobar". ``Dangerous_op'' can be a single operation, or a block
00101  * of code, function calls, or any combination, so long as no alternate
00102  * return is possible before the PyFPE_END_PROTECT macro is reached.
00103  * 
00104  * The macros can only be used in a function context where an error return
00105  * can be recognized as signaling a Python exception. (Generally, most
00106  * functions that return a PyObject * will qualify.)
00107  * 
00108  * Guido's original design suggestion for PyFPE_START_PROTECT and
00109  * PyFPE_END_PROTECT had them open and close a local block, with a locally
00110  * defined jmp_buf and jmp_buf pointer. This would allow recursive nesting
00111  * of the macros. The Ansi C standard makes it clear that such local
00112  * variables need to be declared with the "volatile" type qualifier to keep
00113  * setjmp from corrupting their values. Some current implementations seem
00114  * to be more restrictive. For example, the HPUX man page for setjmp says
00115  * 
00116  *   Upon the return from a setjmp() call caused by a longjmp(), the
00117  *   values of any non-static local variables belonging to the routine
00118  *   from which setjmp() was called are undefined. Code which depends on
00119  *   such values is not guaranteed to be portable.
00120  * 
00121  * I therefore decided on a more limited form of nesting, using a counter
00122  * variable (PyFPE_counter) to keep track of any recursion.  If an exception
00123  * occurs in an ``inner'' pair of macros, the return will apparently
00124  * come from the outermost level.
00125  * 
00126  */
00127 
00128 #ifdef WANT_SIGFPE_HANDLER
00129 #include <signal.h>
00130 #include <setjmp.h>
00131 #include <math.h>
00132 extern jmp_buf PyFPE_jbuf;
00133 extern int PyFPE_counter;
00134 extern double PyFPE_dummy(void *);

00136 #define PyFPE_START_PROTECT(err_string, leave_stmt) \
00137 if (!PyFPE_counter++ && setjmp(PyFPE_jbuf)) { \
00138     PyErr_SetString(PyExc_FloatingPointError, err_string); \
00139     PyFPE_counter = 0; \
00140     leave_stmt; \
00141 }

00143 /*
00144  * This (following) is a heck of a way to decrement a counter. However,
00145  * unless the macro argument is provided, code optimizers will sometimes move
00146  * this statement so that it gets executed *before* the unsafe expression
00147  * which we're trying to protect.  That pretty well messes things up,
00148  * of course.
00149  * 
00150  * If the expression(s) you're trying to protect don't happen to return a
00151  * value, you will need to manufacture a dummy result just to preserve the
00152  * correct ordering of statements.  Note that the macro passes the address
00153  * of its argument (so you need to give it something which is addressable).
00154  * If your expression returns multiple results, pass the last such result
00155  * to PyFPE_END_PROTECT.
00156  * 
00157  * Note that PyFPE_dummy returns a double, which is cast to int.
00158  * This seeming insanity is to tickle the Floating Point Unit (FPU).
00159  * If an exception has occurred in a preceding floating point operation,
00160  * some architectures (notably Intel 80x86) will not deliver the interrupt
00161  * until the *next* floating point operation.  This is painful if you've
00162  * already decremented PyFPE_counter.
00163  */
00164 #define PyFPE_END_PROTECT(v) PyFPE_counter -= (int)PyFPE_dummy(&(v));
